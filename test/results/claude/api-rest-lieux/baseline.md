Voici une vue d'ensemble de l'architecture, puis les détails.

---

## Stack recommandé

**Backend** : Node.js + Fastify (ou Python + FastAPI si la team est plus Python) — les deux offrent validation de schéma native, OpenAPI auto-généré, et des performances adaptées à une API géospatiale.

**Base de données** : PostgreSQL + extension PostGIS, indispensable pour les requêtes géospatiales ("lieux dans un rayon de X km", "lieux dans un polygone"). Prisma ou Drizzle comme ORM.

**Cache** : Redis pour mettre en cache les résultats de recherche fréquents (top lieux par ville, requêtes répétées).

**Auth** : JWT via un middleware, avec rate limiting par token.

---

## Schéma de données principal

```sql
places (id, name, slug, description, category_id, lat, lng, geom [PostGIS],
        address, city, country, phone, website, opening_hours [JSONB],
        price_range, rating_avg, review_count, images [JSONB], created_at, updated_at)

categories (id, name, slug, parent_id)   -- arbre de catégories (musée > art > moderne)

reviews (id, place_id, user_id, rating, body, created_at)

tags (id, name)  +  place_tags (place_id, tag_id)
```

## Endpoints principaux

```
GET    /places                  → liste paginée (filtres: city, category, tags, price_range)
GET    /places/nearby           → ?lat=&lng=&radius_km= (requête PostGIS ST_DWithin)
GET    /places/:id              → détail complet
POST   /places                  → création (admin/partenaire)
PATCH  /places/:id              → mise à jour partielle
DELETE /places/:id              → suppression logique (soft delete)

GET    /places/:id/reviews      → avis paginés
POST   /places/:id/reviews      → poster un avis (authentifié)

GET    /categories              → arbre complet
GET    /categories/:slug/places → lieux d'une catégorie
```

## Points de vigilance

**Géospatial.** L'index PostGIS doit être de type `GiST` sur la colonne `geom` — un index B-tree classique ne fait rien pour les requêtes spatiales. Penser aussi à normaliser les coordonnées en entrée (bbox valide, rejeter les lat/lng aberrantes).

**Pagination.** Préférer la pagination par curseur (`cursor`) plutôt qu'`OFFSET` pour les grandes tables — `OFFSET 10000` scanne 10 000 lignes inutilement. Un curseur basé sur `(created_at, id)` est stable et performant.

**Cache invalidation.** Le cache Redis doit être invalidé à chaque écriture sur un lieu (PATCH/DELETE). Utiliser une clé de type `place:{id}:*` pour pouvoir invalider en pattern, ou adopter une stratégie de TTL court (5–15 min) si l'invalidation fine est complexe.

**Recherche fulltext.** PostgreSQL supporte le fulltext natif avec `tsvector` — suffisant pour commencer. Si le volume ou la qualité de recherche devient critique (fautes, synonymes, ranking), migrer vers Meilisearch qui s'intègre facilement.

**Sécurité des uploads.** Les photos de lieux ne doivent jamais être servies directement depuis le serveur API. Uploader sur S3/R2, valider le type MIME côté serveur (pas juste l'extension), et servir via CDN avec signed URLs à durée limitée.

**Rate limiting différencié.** Les endpoints de lecture (`GET /places/nearby`) peuvent être consommés massivement par des bots. Appliquer des quotas distincts : lectures généreuses (ex. 300 req/min), écritures strictes (10 req/min), et un quota séparé pour les clés API partenaires.

**Versionning.** Préfixer dès le début avec `/v1/` dans l'URL — rétrocompatibilité garantie quand viendra `/v2/`.
